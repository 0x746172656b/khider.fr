Architecture

This site is served from a three-node Kubernetes cluster I run at home, with redundant storage, automated TLS, daily off-site backups, and every component declared in Git. This page shows how the pieces fit together.

The request path

Visitor
https://khider.fr
Cloudflare DNS + reverse proxy · hides the home IP · records managed with Terraform
proxied HTTPS to my home connection
Talos Kubernetes cluster 3 nodes · immutable Linux managed only through an API (no SSH, no shell)
Traefik ingress terminates TLS (certificates automated by cert-manager) and routes each request to the matching application
this website Authentik SSO Jellyfin Prometheus + Grafana Flux Velero
persistent volumes over iSCSI
Synology NAS RAID1 NVMe pool · volumes provisioned by the Synology CSI driver
daily backups · Velero + Kopia
Cloudflare R2 off-site copies of stateful apps · 14-day retention

Cloudflare proxies all traffic, so the cluster's public address is never exposed. Inside the cluster, Traefik routes each request to the matching application. Stateful workloads (databases, application configuration, media libraries) keep their volumes on a mirrored NVMe pool on the NAS, and Velero copies that data to an off-site R2 bucket every night, so a full loss of the local hardware does not mean data loss.

How changes ship

The cluster is not configured by hand: it continuously pulls its desired state from a public Git repository and reconciles itself against it.

Every change is a commit: reviewed, merged, then rolled out automatically. Renovate opens pull requests when a component has an update available, and Terraform applies DNS changes to Cloudflare through the same pull-request flow. The entire lab can be audited or rebuilt from one repository.

When it breaks

Prometheus and Grafana monitor the cluster, including SMART health on the NAS drives. When something fails, I handle it like a production incident and publish a postmortem.

The full manifests, Terraform code, and backup policies are in homelab-infra on GitHub.
